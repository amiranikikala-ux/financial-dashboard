## 💾 საცავის არჩევანი

სამი ვარიანტი გვაქვს, თქვენ აირჩევთ:

### ვარიანტი (ა) — **ჩატის მეხსიერებისივე ChromaDB** 🔵 ჩემი რეკომენდაცია

**რა:** იგივე `ai_vectors/` folder. დაპირებები ცალკე tag-ებით (`kind:promise`, `status:open`) ინახება, ხდება ცოცხალი ძიება + structured filter-ი.

**Pros:**
- ერთი საცავი — ერთი backup
- დაპირება ავტომატურად იქნება ჩართული სემანტიკური ძიებაში ("Alpha-ზე რა პოლიტიკა შევთანხდით?")
- ცოცხალი, უკვე მუშაობს

**Cons:**
- structured query ("ყველა open, overdue 7+ დღე") უფრო სუსტია ვიდრე SQL

### ვარიანტი (ბ) — **ცალკე SQLite ფაილი**

**რა:** ახალი `ai_journal.db` ფაილი (SQLite). Structured table — ზუსტი filter, sort, aggregate.

**Pros:**
- "ვადა-გადაცილებული 7+ დღე, kind=promise" — მილიწამი
- ცხრილად გამოჩენა advantage-ია
- backup მარტივია (ერთი ფაილი)

**Cons:**
- ცალკე storage — 2 საცავი
- სემანტიკური ძიების გარეშე (თუ არ მოვარგებთ)
- მხოლოდ 1-2 დღე დამატებითი სამუშაო

### ვარიანტი (გ) — **ორივე ერთად (dual-write)**

**რა:** ChromaDB-შიც და SQLite-შიც ერთად.

**Pros:**
- ორივე მხარე სრულყოფილად მუშაობს (semantic + structured)

**Cons:**
- Consistency-ის პრობლემა — ორი წყარო, ერთს განახლება, მეორე მოძველდა
- ზედმეტი complexity Sprint 4-ისთვის
